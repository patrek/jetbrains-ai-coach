# VGV Code Review — Design Spec: Provider-backed learning LLM methods

**Target:** `docs/superpowers/specs/2026-06-17-learning-llm-methods-design.md`
**Type:** Design review (no implementation yet)
**Reviewer perspective:** VGV engineering standards + this project's established "minimal divergence" sidecar pattern.

## Summary

The design is fundamentally sound and the right one: a non-vendored `sidecar/src/learning-provider.ts` module is the correct seam, it mirrors the already-merged `generateRule`/`explainOccurrence` pattern faithfully, the patch-0006 growth is consistent with how PR #14 moved methods into `PROVIDER_METHODS`, and the descope list is disciplined. **It is not yet ready to hand to implementation** because of one source-of-truth misattribution that, taken literally, would send the implementer to the wrong file, and a cluster of under-specified response-shape and validator details that — if the implementer fills them in from the prose alone — will produce output the webview silently drops. None of these are architectural; they are precision gaps. Fix them in the spec and this becomes a clean, low-risk slice. Verdict: **needs work (spec edits), then ready.**

The good news for feasibility: every claim about the existing provider plumbing checks out. `runWithProvider`, the `ProviderRun` type, the `{ error: 'llm-unavailable' }` vs `{ error: 'llm-unavailable', reason }` split, the `LLM_UNAVAILABLE_METHODS` set, and the `OVERRIDES` dispatcher all exist exactly as the spec assumes, and the existing test file already establishes the fake-provider injection pattern the new tests will reuse.

## 🔴 Critical — Must Fix Before Merge

- **Spec §"Decisions" / §"Architecture" — wrong source file named for the logic to re-implement.**
  The spec repeatedly says the prompts, JSON parsing, retries, and normalizers are "adapted from upstream `panel-llm.ts`" and frames the re-implementation around that file because it "imports `vscode`." That is only half true and the misleading half is load-bearing. `panel-llm.ts` owns **only** the JSON schemas (`SCHEMA_QUIZ`, etc.), the `parseLlmJson`/`balanceTruncatedJson` repair helpers, and `callLlm`/`callLlmJson` (the streaming + retry loop). The actual **prompt builders and per-method validators/normalizers** — the bulk of what this PR must re-create — live in `sidecar/vendor/webview/panel-request-service.ts` (`handleGenerateLearningQuiz`, `handleGenerateCodeComparison`, `handleGenerateDidYouKnow`, `handleGenerateLearningResources`, `normalizeQuizQuestions`, `buildQuizSystemPrompt`, `buildQuizUserPrompt`, plus the `toText`/`QUIZ_DIFFICULTIES` helpers). An implementer following the spec literally will study the wrong file and miss the real prompt text and validation rules.
  - Why: This is the single biggest risk to "will it produce VGV-quality code." The fidelity of the re-implementation depends on copying the right prompts and the right filter/clamp logic. The spec points at the wrong well.
  - Fix: Correct the references to name `panel-request-service.ts` as the source of prompts + normalizers and `panel-llm.ts` as the source of JSON-repair only. The "imports `vscode`" justification still holds for `panel-llm.ts` (line 8) **and** for `panel-request-service.ts` (it imports `vscode` and calls `vscode.LanguageModelChatMessage` / `vscode.commands`), so the "re-implement, don't import" decision is still correct — but state it about both files.

- **Spec §"Method Behavior" — the webview's required response wrapper keys are never named.**
  The webview calls these methods and reads specific top-level keys off the result: `result.questions` (page-learning.ts:51,66), `result.rounds` (:473,480), `result.facts` (:538,544), and `data.resources` (:648,656). Upstream's handlers explicitly transform the schema's `{ items: [...] }` into `{ questions }` / `{ rounds }` / `{ facts }` / `{ resources }` before `postResponse`. The spec describes each item's fields in detail but **never states that the handler must return these four wrapper keys**, nor that the provider/LLM returns `{ items: [...] }` that must be unwrapped. An implementer could plausibly return `{ items }` or `{ quiz }` and every code path would "succeed" while the UI renders nothing (`result.questions ?? []` → `[]`).
  - Why: A returned shape that is structurally valid but uses the wrong key is the classic silent-failure this layer is prone to, and it is exactly what the webview gate cannot catch. This belongs in the spec as a hard contract, not left to the implementer to rediscover.
  - Fix: Add an explicit response-shape contract per method: `generateLearningQuiz → { questions: QuizQuestion[] }`, `generateCodeComparison → { rounds: CodeComparisonRound[] }`, `generateDidYouKnow → { facts: DidYouKnowFact[] }`, `generateLearningResources → { resources: CachedResource[] }`. Reference the canonical types in `vendor/webview/page-learning-state.ts:69-105`. Add a test assertion (the §Testing section already says "each method returns the expected shape" — make "shape" mean the wrapper key, not just the item fields).

## 🟡 Important — Should Fix

- **Spec §`generateLearningQuiz` — "difficulty clamped … correctIndex from 0 through 3" misstates upstream's actual validation.**
  Upstream does **not** clamp `correctIndex`; `normalizeQuizQuestions` (panel-request-service.ts:271-286) *rejects* any question whose `correctIndex` is not a number in `[0,4)`, and it requires `choices.length === 4` exactly (not "up to" / not clamped). Difficulty is not clamped either — an unknown difficulty falls back to the request difficulty (`fallbackDifficulty`), and topic falls back to `'general'`. The spec's "clamp" verbs invite an implementer to coerce-and-keep instead of filter-and-drop, which changes behavior (a bad question survives with a mangled index instead of being discarded).
  - Why: "Clamp" vs "reject" is a real behavioral fork. Drift from upstream's filter semantics is exactly what the "re-implement faithfully" decision is meant to avoid.
  - Fix: Restate as: require exactly four choices; require numeric `correctIndex` in `[0,4)` (drop the item otherwise); difficulty falls back to the request difficulty when not one of easy/medium/hard; topic defaults to `general`. Mirror the same precision for the other three methods (code-comparison `category` defaults to `readability`, difficulty defaults to the request difficulty; did-you-know `category` defaults to `api`; resources `type` defaults to `'Resource'`, `reason` defaults to `''`).

- **Spec §`generateCodeComparison` — "non-empty title and explanation" is stricter than upstream.**
  Upstream's filter (panel-request-service.ts:447-453) requires `snippetA`/`snippetB` non-empty but only checks `typeof title === 'string'` and `typeof explanation === 'string'` (empty strings pass). The spec tightens this to "non-empty title and explanation."
  - Why: This is a deliberate-looking deviation presented as if it were parity. It may be fine (arguably better), but the spec elsewhere commits to upstream parity, so an undeclared tightening is an inconsistency a reviewer will flag during implementation.
  - Fix: Either call out the tightening explicitly as an intentional improvement with a one-line rationale, or match upstream's `typeof === string` check. Pick one and say which.

- **Spec §`generateLearningResources` / §"Error Handling" — error contract diverges from upstream and the spec doesn't acknowledge it.**
  Upstream's resources handler is the one method that on failure returns `{ resources: [] }` (a success-shaped empty result, panel-request-service.ts:566), not an error. The spec proposes that provider failures return `{ error: 'llm-unavailable', reason: 'bad-output' }` for this method like the others. That is a defensible and arguably better choice (the webview's resources `.catch` now renders `err.message`, per patch-0006's page-learning.ts hunk), but it is a behavioral change from upstream that the spec presents silently.
  - Why: The webview's resources path has two branches — a fulfilled `{ resources: [...] }` (even empty) renders the list, a rejection renders the error message. Returning `llm-unavailable` routes to the rejection branch, which is the intended new behavior, but only because patch-0006 already changed that `.catch`. The spec should connect these two facts so the implementer doesn't "restore" the empty-array fallback and accidentally swallow provider failures into a blank panel.
  - Fix: Add a sentence noting resources intentionally moves from the upstream empty-array fallback to the `llm-unavailable` failure contract, consistent with the patch-0006 `.catch` that surfaces `err.message`. Keep the "return empty `resources` array when the provider returns zero *valid* resources" rule (that is the success-with-no-items case and is correct).

- **Spec §"Data Flow" step 7 + §"Error Handling" — JSON repair/retry is described as new logic when a vendored, battle-tested helper already exists.**
  The spec says the JSON helper "repairs common JSON formatting issues and retries with a JSON-only nudge." `panel-llm.ts` already contains `parseLlmJson` (fence stripping, JSONL handling, trailing-comma/smart-quote/control-char repair) and `balanceTruncatedJson` (closes truncated output). These are pure, `vscode`-free functions. The spec's "keep prompts, JSON parsing, retries, and normalizers in sidecar-owned code rather than importing panel-llm.ts because that file imports vscode" over-rotates: the *file* imports vscode at module top (line 8), but the *repair functions themselves* are not exported and are vscode-free. The design should decide deliberately: copy these two helpers verbatim into `learning-provider.ts` (faithful, but a second copy that can rot), or extract them to a vscode-free shared module both can import.
  - Why: Re-deriving JSON-repair heuristics by hand is precisely where "VGV-quality" erodes — the upstream version handles JSONL, truncation, and smart quotes that an ad-hoc reimplementation will miss, and getting it subtly wrong yields `bad-output` failures that look like provider problems. This is a real drift-vs-duplication decision the spec should make on the record, matching the same reasoning the rpc-handlers.ts header already documents for the handler map.
  - Fix: State explicitly whether the implementation copies `parseLlmJson`/`balanceTruncatedJson` verbatim (with a comment pointing at the upstream origin, as `explainOccurrence` does for its prompt) or extracts them. Given the project's "minimal divergence" rule, verbatim copy with an origin comment is the most consistent choice — but say so.

- **Spec §"Method Behavior" — the `provider.run()` single-flat-prompt constraint and the structured-output gap aren't addressed.**
  Upstream relies on `callLlmJson` passing a JSON `response_format` schema (`structuredOutputOptions`) to the model, which materially improves JSON validity. `CliProvider.run(prompt, opts)` takes one flat string and has no structured-output channel (cli-provider.ts:50-54). The spec inherits `generateRule`'s "collapse system+user into one prompt" approach (correct) but is silent on the loss of schema-enforced output, which is exactly why the JSON-repair/retry path matters more here than upstream.
  - Why: This is the load-bearing reason the design needs robust JSON repair and a retry nudge. Making it explicit justifies the retry budget and sets the implementer's expectations (CLI text output is messier than schema-constrained API output).
  - Fix: One sentence under Architecture or Method Behavior: the schemas in `panel-llm.ts` cannot be passed to a CLI provider (flat-prompt contract), so each prompt must inline an explicit JSON shape instruction (upstream's prompts already end with a `Respond with a JSON object: {"items":[...]}` line — preserve it), and the repair+retry path compensates for the absence of structured output.

- **Spec §"Testing" — coverage list is good but omits the wrapper-key and `{ items }`-unwrap assertions, and the retry-budget specifics.**
  The list covers shape-on-success, bare/`reason` unavailable, malformed-JSON retry, filtering/clamping, and the `https://` reject. It does not pin: (a) the exact wrapper key per method (see Critical #2), (b) that an `{ items: [...] }` payload is unwrapped, (c) that an array-root payload (`[...]`) is also accepted (upstream handles both: `Array.isArray(response) ? response : response.items`), or (d) the retry count / `MAX_ATTEMPTS` parity with `generateRule` (which uses 2 + 1 final). The existing `rpc-handlers.test.ts` `generateRule` suite is the template — it asserts call counts (`toHaveBeenCalledTimes`) and inspects `fakeRun.mock.calls[1][0]` for the retry-prompt content; the learning tests should match that rigor.
  - Why: VGV's bar is behavior-covering tests, not happy-path. The array-root-vs-`items` branch and the unwrap are the two most likely places a faithful reimplementation diverges silently.
  - Fix: Add to the required-coverage list: unwraps `{ items }` to the method's wrapper key; accepts an array-root payload; retry fires exactly once on malformed JSON then succeeds (assert `fakeRun` call count, mirroring the generateRule tests); the retry prompt contains the JSON-only instruction.

## 🔵 Suggestions — Nice to Have

- **Naming.** `sidecar/src/learning-provider.ts` is good and passes the 5-second rule. Keep the exported handler names identical to the RPC method names (`generateLearningQuiz`, …) so the `OVERRIDES` map reads as a 1:1 table, exactly like the existing entries. Consider a `LEARNING_HANDLERS` aggregate export (mirroring `SDLC_CATALOG_HANDLERS` spread into `OVERRIDES`) so rpc-handlers.ts grows by one spread line rather than four — consistent with the established pattern and keeps the dispatcher tidy.

- **Spec §Testing — name the helper-test file.** The spec says "add focused helper tests if the JSON parsing/normalization logic grows beyond simple inline coverage." If the JSON-repair helpers are copied (Important #4), they deserve their own `learning-provider.test.ts` regardless of size, since they are pure and cheap to test exhaustively (JSONL, trailing commas, truncation, smart quotes). Pin the filename so it isn't skipped.

- **Spec §"Decisions" — the `LLM_UNAVAILABLE_METHODS.size` 10→6 claim is correct; cross-reference the existing test.** The current `rpc-handlers.test.ts:95` already asserts `size === 10` and that `compileNlRule` stays. The spec's "drops from 10 to 6" and "four Learning methods no longer in the set" is consistent and `compileNlRule` correctly remains (its LLM call is buried in `rule-compiler.ts`, as the rpc-handlers.ts header notes). Add a one-line pointer to the existing assertion so the implementer updates it rather than writing a parallel one.

- **Consistency with `generateRule`'s retry idiom.** `generateRule` re-invokes the provider with an augmented prompt embedding the prior attempt + issues, and returns the last attempt even if imperfect. The learning methods differ: they should *filter* bad items rather than return imperfect output, and the retry is a JSON-only nudge (parse failure), not a validation-issue feedback loop. This is a legitimate difference (data extraction vs. document generation) but worth one sentence so a reviewer doesn't expect the generateRule idiom verbatim.

## Simplicity Assessment

- **Lines that could be removed:** none in the design — the scope is already minimal and the descope list (`compileNlRule`, skills, triage, GitHub/SDLC, host push, live URL validation, new providers) is disciplined and explicit.
- **Unnecessary abstractions:** none proposed. One module, four handlers, one local JSON helper. Appropriately right-sized — this is not a premature `BaseLearningProvider<T>` situation.
- **YAGNI violations:** none. The "conservative `https://`-only, no live verification" decision actively resists scope creep and is the correct call.
- **Complexity verdict:** Already minimal. The only complexity question is drift-vs-duplication on the JSON-repair helpers (Important #4) — and the project's own "minimal divergence" precedent (documented in rpc-handlers.ts) favors a verbatim copy with an origin comment over a hand-rolled reimplementation.

## Testing Assessment

- **New code with tests:** Planned and credible. The spec extends `rpc-handlers.test.ts` and reuses the established fake-provider injection (`vi.mock('./cli-provider', …)` with a `fake` id) — this pattern already exists and works, so the test approach is feasible as written.
- **Test quality:** The proposed list avoids the happy-path-only trap (covers absent provider, provider failure + reason, malformed JSON, invalid-item filtering, URL rejection). Gaps: wrapper-key assertion, `{ items }`-unwrap, array-root acceptance, and explicit retry-count assertions (see Important #6). With those added it meets the VGV bar.
- **State management test coverage:** N/A — this is sidecar RPC handler logic, not UI state. The webview gating is covered by the patch-0006 row in the required-coverage list ("webview gating treats all four Learning methods as provider-backed"); ensure that test actually exercises `PROVIDER_METHODS` membership, not just `LLM_METHODS`.
- **Verification target:** `cd sidecar && npm test` is correct and matches the existing suite's runner (vitest).

## Pattern-Adherence Verdict (the questions asked)

- **Is `sidecar/src/learning-provider.ts` the right seam?** Yes. Non-vendored, vscode-free, layered on the existing `cli-provider` path, dispatched through `OVERRIDES` exactly like `generateRule`/`explainOccurrence`. Correct.
- **Is "re-implement rather than import" justified?** Yes, but for the right reason: both `panel-llm.ts` and `panel-request-service.ts` import `vscode` at module scope and `callLlmJson` is intrinsically bound to `vscode.lm` / `LanguageModelChatMessage`, so they cannot be imported into the sidecar. The risk of drift is real and concentrated in the JSON-repair helpers — mitigate by copying them verbatim with an origin comment (Important #4), consistent with how `explainOccurrence` copies its prompt verbatim.
- **Is the patch-0006 growth consistent with PR #14?** Yes. Moving the four methods into `PROVIDER_METHODS` while keeping them in `LLM_METHODS` for timeout handling is exactly the shape PR #14 established for `generateRule`/`explainOccurrence` (they appear in both sets), and the `capabilities.provider.status` preflight already exists in `shared.ts`. The patch grows by adding four strings to one existing set — minimal and consistent.
- **Are the per-method response shapes/validators correctly specified?** Partially. The *item* fields are accurate against `panel-llm.ts` schemas and `page-learning-state.ts` types, but (a) the wrapper keys are unnamed (Critical #2), (b) several validators say "clamp" where upstream "rejects" (Important #1), and (c) one tightening and one error-contract change are undeclared (Important #2, #3). Fix these and the validators will match the webview's expectations precisely.
