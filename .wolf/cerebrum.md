## Do-Not-Repeat (2026-06-15)
- **Sidecar main.js changes silently don't reach the IDE if runtime extraction is stale.** The sidecar bundle is extracted to `~/.ai-coach-jetbrains/runtime/<ver>/` and run from there; the webview `app.js` is served straight from the JAR by `AssetSchemeHandler`. So a webview change loads immediately but a `sidecar/` change only loads if `SidecarRuntime.ensureExtracted` actually re-extracts. The runIde sandbox jars the plugin (`jar://`), and the version dir name never changes in dev — so any version/`isComplete`/protocol-based reuse check strands the first extraction forever. Freshness MUST key on bundle content (SHA-256 fingerprint in `.bundle-hash`), not the version string or classpath protocol. When verifying a sidecar fix in the IDE, check `grep -c <symbol> ~/.ai-coach-jetbrains/runtime/*/main.js` — not just `sidecar/dist`.
- **Bumping the persisted Session/parse shape requires bumping `CACHE_VERSION`** (sidecar/vendor/core/cache.ts) or the IDE reuses a stale `~/.ai-coach-jetbrains/cache/parsed.json` that lacks the new field.

## Decision Log (part 4 — theme integration, 2026-06-15)
- **Webview state uses a single `aicoach.webviewState` blob key, NOT per-page `aicoach.webviewState.<page>`.** The part-4 plan mentioned `<page>` keys, but the part-3 `getState`/`setState` bootstrap shim (and upstream VS Code's state API) round-trips one whole-state object. Single key is correct; the plan wording was an over-specification. Do not "fix" it by splitting per page.
- **The webview persists ONLY learning/budgets/experiments/achievements via `getState`/`setState` — NEVER the current page or harness/workspace filter (verified in IDE 2026-06-15).** `setState` calls live in page-achievements (`achievementState`), page-burndown (`modelBudgets`), page-experiments (`experiments`/`activeLabTab`), page-learning-state (`learningState`). The active page + filters are module-level memory (`page-dashboard.ts` "Module-level view state"). So: page/filter survive tool-window hide/show (IntelliJ keeps the JS context alive, matching VS Code `retainContextWhenHidden`) but RESET on full IDE restart — this is correct upstream parity, NOT a bug. The plan's acceptance criterion "page/filter survives IDE restart" was wrong about what the webview persists. The Kotlin round-trip itself works (the getState blob is written to the project's `.idea/workspace.xml` under `PropertiesComponent` and read back via `WebviewBridge.currentState` → `__INITIAL_STATE__`). Don't try to "fix" restart page/filter loss without an explicit decision to add host-side state beyond upstream.
- **Theme switch = live `:root` set-property inject ONLY — NO reload (corrected 2026-06-15, see buglog bug-030).** The soft-reload approach (originally chosen to recolor charts) is BROKEN: the webview renders only after the sidecar's one-shot `dataReady` push, which fires once per sidecar *connection*. A soft reload restarts app.js on the same bridge/connection, so `dataReady` is never re-sent and the reloaded page hangs forever on "Building Activity Index". (A tool-window close/reopen makes a NEW connection → new handshake → new `dataReady` → works; only the in-place reload hangs.) Fix: live CSS-var injection recolors the whole CSS UI instantly with no reload. The bundled Chart.js `Chart` is module-local in vendored app.js (not on `window`, vendored files must not be patched), so mounted charts keep their palette until the tool window is reopened — accepted trade-off. **Never reintroduce a same-connection `reload()` in the webview without first re-triggering `dataReady`.**
- **`WebviewThemeSync` is deliberately untested.** It's a thin JCEF/EDT/message-bus class (one LafManagerListener subscription + two browser calls + a load gate), verified manually per plan tasks 3-5. Adding a browser-seam abstraction solely to unit-test 3 lines would violate the no-premature-abstraction rule. `ThemeCssProvider` (the pure logic) carries the unit tests instead.
- **`test-harness.html` shipping in the production JAR is a known deferral to part 7.** A ~25 KB upstream Playwright artifact is served at the webview origin (committed in part 3, wired in build.gradle.kts processResources/resources). Out of part-4 scope; part 7 (hardening/packaging/Marketplace) is where shipped-JAR test artifacts get stripped. Revisit then.

## Decision Log (part 5 — trust gate + project scoping, 2026-06-15)
- **The trust GATE was never enforced before part 5 — a real security gap, not just missing UI.** `sidecar/vendor/core/detector-registry.ts` calls `loadPersonalRules()` UNGATED at module import, and `setDefaultTrustGate`/`loadProjectRules` were never called anywhere. So personal rules executed without approval. Part 5's `sidecar/src/rule-scope.ts` `ruleScope.install()` (called in `rpc-server.start()` after `installTrustMemento`, before `loadData`) installs `createTrustGate(store)` and re-loads personal rules gated before any request is dispatched. Don't assume the sidecar enforces trust without this install step.
- **Approval is SIDECAR-DRIVEN, not Kotlin-driven (user-confirmed).** The dialog sends chosen filePaths; the sidecar's `approveLocalRules` calls vendored `approve(store, path, pendingEntry.content)` — hashing the AS-LOADED content, never a re-read. This is the TOCTOU guard and keeps `rule-trust.ts` authoritative (D5). The Kotlin `TrustStoreService` is a dumb generic key→JSON memento; it records NO hashes. Don't move hashing host-side.
- **Per-request project scoping = a promise-chain mutex in the sidecar (user-confirmed "serialized reload-on-change").** `RuleScope.run()` wraps EVERY dispatch; it reloads the project layer only when the request's effective root (projectRoot unless safeMode) differs from what's loaded. One shared sidecar serves all windows, so this prevents project A's rules executing on project B. Handlers run INSIDE `run` — they must call the free functions / `reloadCurrent` directly, NEVER `ruleScope.run` (would deadlock the chain).
- **`loadAllMetricLayers()` MUST NOT be called more than once.** It does `clearMetrics()` then guarded `registerAllBuiltinMetrics()`; the one-time guard means the 2nd call drops built-in metrics permanently. `rule-scope.ts` snapshots built-in metrics once and re-seeds them by hand on every reload (builtin→personal→project order). See buglog.
- **safe-mode hard-blocks the project layer.** The bridge stamps `safeMode = !project.isTrusted()` (`com.intellij.ide.impl.isTrusted` extension fun) on every forwarded envelope AND every `hostCall`; the sidecar drops the project layer entirely when safeMode. Untrusted IntelliJ projects load no project rules/metrics regardless of approval.
- **`LocalRuleWatcher` and `TrustGateController`'s JCEF/EDT/notification glue are deliberately untested** (same precedent as `WebviewThemeSync`, part 4). The pure logic is extracted and unit-tested instead: `parsePendingRules` (top-level fun in TrustGateController.kt → `TrustGateParsingTest`), `safeMode` stamping + `hostCall` routing/fail-fast (`SidecarSupervisorTest`), and the gate/scoping/TOCTOU behaviors (`sidecar/test/trust-gate.test.ts`). Don't add a JCEF seam abstraction just to unit-test the thin glue.
- **trust host-requests delegate to an injected `TrustStore` interface; `hostCall` is a new host-originated request path.** `SidecarSupervisor` stays IntelliJ-free: `TrustStore` (default `EmptyTrustStore`) is injected, real binding in `SidecarService` → `TrustStoreService`. `hostCall(method, params, projectRoot, safeMode, onResult)` lets the host (trust dialog) call sidecar methods and consume the response in Kotlin (separate `hostCalls` correlation map, prefix `k`); callbacks run on the supervisor stream thread → marshal to EDT.

## Decision Log (2026-06-15)
- **Skill Finder's two errors are accepted deferrals, NOT bugs — do not "fix" them.** `Error: llm-unavailable` (Custom Skill Opportunities) is the ADR 0006 degrade of `triageSkills` because the JetBrains host has `llm:false`. `Catalog error: Unknown method: discoverCatalog` (Community Skills) is the part-6 catalog port per ADR 0009 (`discoverCatalog`/`installSkill`/`installCatalogItem`). The page (`vendor/webview/page-skills.ts`) has no capability gating and surfaces both raw. User decided 2026-06-15 to leave as-is and defer graceful degradation + the catalog implementation to part 6. **RESOLVED in part 6 (2026-06-16):** both deferrals are now done. The catalog/SDLC methods are ported in `sidecar/src/sdlc-catalog.ts` (filesystem/network only, vscode→Node fs), wired into `rpc-handlers.ts` OVERRIDES. LLM degradation is the single sanctioned webview patch `tools/patches/0006-webview-llm-capability-gate.patch` — `shared.ts` gates the 12 LLM methods on `getCapabilities().llm===false` (proactive) AND translates `llm-unavailable` into a friendly MCP-pointing message (reactive). So even if the proactive `LLM_METHODS` set in shared.ts drifts from `LLM_UNAVAILABLE_METHODS`, the reactive translation still prevents a hang — drift is a missed optimization, not a correctness bug.

## Decision Log (part 6 — MCP server, ports, export, degradation, 2026-06-16)
- **The standalone MCP server (`sidecar/src/mcp-main.ts` → `mcp-server.ts` → `mcp-tools.ts`) must stay IDE-free and stdout-clean.** stdout is the MCP protocol channel, so `mcp-main.ts` redirects `console.log/info/debug` to stderr BEFORE dynamic-importing the heavy graph (mirrors `main.ts`'s env-stamp-then-dynamic-import dance), and `mcp-server.ts` SKIPS `analyzer.warmUp()` (it runs in a worker_thread that inherits this process's stdout → would corrupt the protocol; formatters don't need it). Cache-first: serve `loadCacheData()` (or empty) instantly, `void this.refresh()` in the background, prepend `partialDataNote` while `isParsing`. Headless trust: `installTrustMemento(createHostTrustMemento())` with NO channel → empty store → untrusted rules excluded (built-ins only). Used the low-level SDK `Server` + raw JSON-Schema `inputSchema` (no zod) to avoid zod-version coupling.
- **`vendor/mcp/{formatters.ts,tools.ts}` are vendored byte-identical (manifest `src/mcp`→`sidecar/vendor/mcp`).** `formatters.ts` is vscode-free and REUSED; `tools.ts` is vscode-coupled and UNUSED (the 12-tool table is re-derived in `mcp-tools.ts` as our own code). Tool names pinned `aiEngineerCoach_*` (ADR 0002).
- **`runtime/current/` is a copy-mirror, not a symlink** (`SidecarRuntime.mirrorToCurrent`), guarded by the bundle fingerprint — for Windows-without-symlink-privilege + IDE-closed support. The MCP client config (`node ~/.ai-coach-jetbrains/runtime/current/mcp-main.js`) points here so it survives plugin updates. `mcp-main.js` had to be added to `processResources` include list AND to `REQUIRED`.
- **`exportSummary` is host-owned (never forwarded).** The bridge intercepts it and fetches rendered content from the sidecar via `hostCall('exportSummaryContent', …)` (a NEW internal sidecar method, distinct from the webview-facing `exportSummary`), then runs the IntelliJ folder chooser on the EDT and writes date-stamped files. Webview only sees `{ok|cancelled|error}`.
- **Local dev gotcha:** the vendored `antipatterns-e2e.test.ts` ("loads all 45 built-in rules") fails if `~/.ai-engineer-coach/rules/` contains a stray personal rule (detector-registry loads it ungated at import → 46). CI has a clean home. Not a code bug.
- **Degradation message framing (user-confirmed 2026-06-16):** the LLM-unavailable message must NOT say "use the MCP tools (aiEngineerCoach_*)" for generation features — the 12 MCP tools are read-only ANALYTICS and have NO generation equivalent (no quiz/slop/rule-gen tool). Correct split: generation features → "ask Claude Code to do it directly"; analytics → MCP tools. Current text in `shared.ts` (patch 0006): *"This needs a language model, which the JetBrains plugin doesn't include — ask Claude Code to do it directly. (Your usage analytics are available there too, via the AI Engineer Coach MCP tools.)"* The Learning page's "Did you know"/Resources catch blocks were also patched (0006 now covers `page-learning.ts`) to surface `err.message` instead of swallowing it, so the whole Learning page degrades consistently.
- **Patch generation vs RTK:** RTK rewrites `git show`/`diff` output (summarizes it), corrupting pristine-blob extraction for patch regen. Generate vendor patches with a Node script using `execFileSync('git', …)` (NOT intercepted by the Bash hook), or `rtk proxy` — never a bare `git show > file` in the shell. See `/tmp/genpatch.mjs` pattern: extract pristine blob per file, `git diff --no-index a/… b/…` from a temp tree, then strip the doubled `a/a/`→`a/` prefix.
- **Known deferred (part 7):** the Quiz/"Slop or Not" error still shows a Retry button that can't succeed without an LLM (vendored; hiding it needs more webview divergence). `bug-060`: runIde on JBR 25 throws once at SidecarProcess.kt:31 (env, self-recovers).

## Decision Log (part 7 — hardening, packaging, branding, 2026-06-16)
- **Marketplace display name is "AI Usage Coach"; the plugin id `com.aicoach.jetbrains`, the `aicoach`/`com.aicoach` package paths, and the MCP tool names `aiEngineerCoach_*` stay fixed.** The rename is display-only (plugin.xml `<name>`/`<vendor>`, toolWindow id, configurable displayName, notification titles, README, index.html `<title>`, user-facing error prose). Tool names are pinned by ADR 0002 for client-config stability; the id can't change without breaking installed users.
- **Directory exclusion is a privacy "never read" control with TWO seams (the discovery is split):** `~/.copilot` flows through the sidecar-OWNED `findLogsDirs()` call sites (`rpc-server.ts`/`mcp-server.ts`) → filtered by `src/dir-exclusion.ts`, NO patch. Claude/Codex/OpenCode flow through vendored `parser-harnesses.ts` (`findClaudeDirs/findCodexDirs/findOpenCodeDirs` called internally) → needs sanctioned patch `0007-dir-exclusion-env.patch`, which INLINES the match logic (vendored tree must not import out of `vendor/`). Contract: Kotlin `CoachSettings.excludedDirs` → `AI_COACH_EXCLUDED_DIRS` env (platform-path-separator joined) set on the sidecar process; the parse worker inherits `process.env` (fork/Worker omit explicit `env`), so the env reaches it. A dir is excluded if it equals or is nested under an excluded root.
- **The exclusion setting applies LIVE, not just on restart (vgv-review caught this).** The app-level sidecar is a singleton; `ensureStarted` is a no-op after first launch, and the env is fixed at process spawn. `CoachSettingsConfigurable.apply()` calls `SidecarService.reloadExcludedDirs()`, which re-snapshots the setting into the `@Volatile launch` record and calls `supervisor.requestRestart()` (terminates + relaunches → new process reads the new env). Without this, excluding `~/.claude` would silently not take effect until IDE restart — contradicting the disclosure text.
- **`excludeScannedDirs` (the inlined patch-0007 copy) is covered by a SIDECAR-OWNED test** (`sidecar/test/dir-exclusion-harness.test.ts`), NOT a vendored one — the vendored `parser-harnesses.test.ts` is overwritten on re-sync. The test drives the exported `hasExternalHarnessSources()` with a temp HOME + `AI_COACH_EXCLUDED_DIRS` since `excludeScannedDirs` is not exported.
- **`test-harness.html` (the part-4 deferral) is REMOVED, not just unshipped.** It was a ~25 KB upstream Playwright artifact committed in part 3 with no Playwright runner wired in this repo. Deleted the resource + its `AssetSchemeHandler` route (no longer in the production JAR). If a Playwright harness is wanted later, that's net-new work.
- **Latent bug fixed (`bug-080`):** `McpDiscoveryNotifier`/`ExportSummaryHandler` used notification group id `"AI Coach"`, never registered in plugin.xml (only `"AI Coach Trust"` was) → those balloons silently never fired. Part-7 registers all groups under the new brand and aligns the GROUP_ID constants.
- **`verifyPlugin` spans the 5-product matrix (IDEA CE, PyCharm CE, WebStorm, GoLand, Rider, all 2024.2.5 stable) but MUST run one IDE per process — see `bug-081`.** The verifier shares ONE cached NIO zip FileSystem for the plugin jar across its per-IDE threads; verifying several IDEs in a single JVM races on that FileSystem being closed by the first thread to finish → `java.nio.file.ClosedFileSystemException` + a cascade of `ClosedByInterruptException` reading IDE jars. Fix: `build.gradle.kts` reads `-PverifyIdes=<keys>` (comma-separated IC/PC/WS/GO/RD; default `all` locally) to restrict the `pluginVerification.ides {}` set, and the CI `plugin` job is a `matrix: { ide: [IC,PC,WS,GO,RD] }` running `verifyPlugin -PverifyIdes=${{matrix.ide}}` — one IDE per job (also keeps each job to one IDE of disk, so the original ~17 GB free-disk is enough; do NOT delete `/opt/hostedtoolcache`, that's where setup-java/node provision). Platform-free `:plugin:test` runs once, gated to the IC shard. Signing/publishing creds come from env only (`CERTIFICATE_CHAIN`/`PRIVATE_KEY`/`PRIVATE_KEY_PASSWORD`/`PUBLISH_TOKEN`); `buildPlugin` works without them.
